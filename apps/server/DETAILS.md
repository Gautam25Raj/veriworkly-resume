## Architecture Overview

The server follows a **layered architecture** with clear separation of concerns:

```
Incoming Request (HTTP)
    ↓
[CORS Middleware] → [Rate Limiting] → [Logging] → [Auth Check]
    ↓
[Express Route Handler] → [Controller] → [Service] → [Data Access]
    ↓
[Prisma ORM] → [PostgreSQL] or [Redis Cache]
    ↓
[Background Jobs via BullMQ] → [Playwright Rendering] → [S3/Local Storage]
```

### Middleware Pipeline

All requests flow through this middleware stack in order:

1. **Helmet** - Security headers (CSP, X-Frame-Options, HSTS, etc.)
2. **CORS** - Origin validation with credentials support
3. **Rate Limiting** - Redis-backed sliding window (100 req/15min on public endpoints)
4. **Logging** - Structured JSON logging with Pino (request/response metadata)
5. **Body Parser** - JSON and URL-encoded parsing (4MB limit)
6. **Trust Proxy** - Configurable for reverse proxy headers (X-Forwarded-For, etc.)
7. **Auth Diagnostics** - Logs auth request details for debugging
8. **Route Handlers** - Versioned API routes under /api/v1/*

Error handling is centralized in a global error handler middleware that catches unhandled exceptions and formats consistent error responses with HTTP status codes and error details.

### Request Handling Pattern

Controllers parse and validate input, delegate to services, and format responses. Services contain pure business logic. Data access goes through Prisma for databases or direct Redis calls for caching.

Example flow for resume export:
1. Client POST /api/v1/exports with resume snapshot and desired format (PDF/PNG/JPG)
2. Controller validates schema with Zod
3. Service enqueues job to BullMQ (Redis-backed queue)
4. Returns job ID immediately (async processing)
5. Client polls GET /api/v1/exports/:jobId for status
6. When complete, client downloads via GET /api/v1/exports/:jobId/download
7. Background worker processes job after slot becomes available

## Authentication System (Better-Auth)

Better-Auth is a modern, modular authentication framework that replaces traditional passport.js approaches. The server implements passwordless-first authentication:

**Authentication Methods**
- Email OTP (One-Time Password) - 6-digit codes sent via Nodemailer
- Magic Links - Email-based passwordless sign-in
- OAuth Providers - Google and GitHub OAuth2 flows
- Session-based authentication with database persistence

**Session Management**
- Sessions stored in PostgreSQL via Prisma (not stateless JWT)
- Session TTL: 30 days by default (configurable via AUTH_SESSION_TTL_SECONDS)
- Session reset on use: 24 hours (configurable)
- Optional JWT caching strategy for production to reduce database queries
- Cookie-based transport with secure, httpOnly, sameSite=Lax flags
- Cross-domain cookies via cross-subdomains feature for .veriworkly.com

**Request Flow**
1. Client sends email address to /api/v1/auth/sign-in/email
2. Better-Auth sends OTP email via Nodemailer SMTP
3. Client enters OTP code and submits to /api/v1/auth/verify-email
4. Server validates OTP (6-digit, 15-minute expiry, max 5 attempts)
5. Session created and cookie set in response
6. Subsequent requests include session cookie, auto-validated by Better-Auth middleware

**Configuration**
- AUTH_SECRET - Cryptographic key for session encryption (min 32 characters)
- AUTH_BASE_URL - Server URL for building OAuth redirect URIs
- AUTH_IP_ADDRESS_HEADERS - Headers to extract client IP (x-client-ip, x-forwarded-for, etc.)
- Trusted origins list determines which frontends can authenticate

## PDF Export Workflow: From React UI to ATS-Compliant Document

The PDF export is the most complex workflow, involving client snapshot capture, server queuing, headless rendering, and artifact storage.

### Client Initiation (Resume Builder UI)

The resume builder frontend renders React components based on template type (modern, minimal, ats-classic, executive). When user clicks "Export as PDF":

1. **Capture DOM Snapshot** - JavaScript extracts resume data from Zustand store (local state)
2. **Generate renderHtml** - Optionally renders the visible DOM to HTML string using tools like html2canvas or custom DOM serialization (or uses pre-built server fallback)
3. **Send to Server** - POST /api/v1/exports with:
   ```json
   {
     "format": "pdf",
     "snapshot": { ... full resume JSON object ... },
     "renderHtml": "<html>...</html>" (optional, pre-rendered template HTML)
   }
   ```
4. **Receive Job ID** - Server responds with export job ID (polling endpoint)

### Server-Side Queueing (exportQueueService.ts)

The export controller receives the request and validates input:

```javascript
const resumeExportSchema = z.object({
  format: z.enum(["pdf", "png", "jpg"]),
  snapshot: z.record(z.any()),
  renderHtml: z.string().min(1).max(3_000_000).optional()
});
```

**Key Validation Rules:**
- renderHtml payload max 3MB (large DOM captures)
- Format must be pdf, png, or jpg
- Snapshot is flexible JSON (resume data)

**Cache Checking:**
- Generates cache key: SHA256(format + snapshot + renderHtml)
- Checks Redis for existing rendered artifact (cache-aside pattern)
- If cache hit, returns artifact immediately without re-rendering
- Prevents redundant Playwright renders for identical requests

**Job Enqueueing:**
1. Creates BullMQ job with data: `{ ownerType, userId, resumeId, snapshot, format, fileNameBase, renderHtml }`
2. Sets retention: job kept for 24 hours (configurable)
3. Returns job ID and status "queued"
4. Client polls GET /api/v1/exports/:jobId for status updates

### Export Worker Processing (BullMQ Worker)

The background worker starts via `startExportQueueWorker()` and maintains a pool of concurrent renders:

**Concurrency Control:**
- Max concurrent exports: 4 by default (configurable)
- Semaphore-like pattern using `activeRenderCount` and `pendingRenderResolvers`
- Workers wait for available render slot before proceeding
- Prevents memory exhaustion from too many Playwright instances

**Job Processing:**
1. Worker dequeues job from Redis queue
2. Updates job status to "processing"
3. Acquires render slot (waits if at max concurrency)
4. Calls `exportResumeSnapshot(snapshot, format, renderHtml)`
5. On success: stores artifact, updates cache, sets status to "completed"
6. On error: sets status to "failed" with error message
7. Releases render slot for next job

## Playwright Headless Rendering Pipeline

This is where the magic happens. `exportResumeSnapshot()` in exportService.ts orchestrates Playwright to render perfect resumes.

### Browser Lifecycle Management

```javascript
let browserInstance: Browser | null = null;

async function getBrowser() {
  if (browserInstance) return browserInstance;
  
  browserInstance = await chromium.launch({
    headless: true,
    args: ["--no-sandbox", "--disable-setuid-sandbox"]
  });
  
  browserInstance.on("disconnected", () => {
    browserInstance = null; // Restart on unexpected disconnection
  });
  
  return browserInstance;
}
```

**Browser Reuse:** Single Chromium instance shared across all render jobs (connection pooling). Automatically relaunches if crashed.

**Security:** --no-sandbox for Docker/container environments, --disable-setuid-sandbox for privilege separation.

### Page Rendering Process

For each export:

```javascript
const context = await browser.newContext({
  viewport: { width: 1280, height: 1800 },
  javaScriptEnabled: false // Critical: Deterministic rendering
});

const page = await context.newPage();
await page.emulateMedia({ media: "screen" });

await page.setContent(renderHtml || buildHtml(snapshot), {
  waitUntil: "networkidle",
  timeout: 30_000
});
```

**Viewport Size:** 1280x1800px approximates A4 paper size (210x297mm) at screen DPI.

**JavaScript Disabled:** Ensures deterministic rendering (no random values, no async data loads). Resume content must be fully static HTML.

**Content Injection:**
- Uses provided `renderHtml` if sent from client (rendered React DOM)
- Falls back to `buildHtml(snapshot)` - server-generated semantic HTML from resume data

**Network Idle:** Waits for font downloads to complete (critical for web fonts like Geist, Merriweather, Manrope).

### Font Loading & CSS Variable Resolution

The biggest gotcha in PDF export parity is fonts. The server must load fonts identically to the frontend editor.

**Font Resolution:**
1. Resume customization includes `fontFamily` field: "geist", "serif", or "modern"
2. Maps to CSS font stack and Google Fonts stylesheet:
   - geist: "Geist", "Inter", "Segoe UI", Arial, sans-serif + Google Fonts URL
   - serif: "Merriweather", Georgia, Cambria, serif + Google Fonts URL
   - modern: "Manrope", "Segoe UI", "Helvetica Neue", Arial, sans-serif + Google Fonts URL

**CSS Variable Injection:**
```javascript
const { cssFontFamily, stylesheetHref } = resolveSnapshotFontFamily(snapshot);

// Injected into <style> tag:
body { font-family: ${cssFontFamily}; }
```

**Font Readiness Polling:**
```javascript
await page.evaluate(async () => {
  const fontsReady = document.fonts?.ready;
  if (fontsReady) {
    await fontsReady; // Waits for all @font-face declarations to load
  }
});
```

**Critical Issue (Documented in repo memory):**
- If server export misses inherited CSS font variables (especially custom properties like --font-resume-serif), rendered fonts diverge from preview
- Solution: Always include known font CSS stacks in build-export-html and inject stylesheet links
- Body parser limit must be >=3MB to handle large renderHtml payloads

### PDF Generation with Print Media

After fonts load, Playwright renders to PDF:

```javascript
if (format === "pdf") {
  const buffer = await page.pdf({
    format: "A4",
    printBackground: true,
    preferCSSPageSize: true,
    margin: { top: "0mm", right: "0mm", bottom: "0mm", left: "0mm" }
  });
}
```

**Print Mode Considerations:**
- Chromium's print CSS is applied (media: "print" not explicitly set, but behaves as print context)
- printBackground: true ensures colors and backgrounds render in PDF
- preferCSSPageSize respects CSS page size rules (@page { size: A4; })
- Zero margins: Resume HTML defines its own spacing

**ATS Optimization:**
- No embedded fonts (creates larger PDFs, slower to parse)
- Uses system fonts as fallback
- Simple semantic HTML structure (no complex CSS grid in PDFs)
- Black text on white background maximizes ATS parser compatibility
- No images, no JavaScript artifacts, no interactive elements

### Image Export (PNG/JPG)

```javascript
if (format === "png" || format === "jpg") {
  const screenshot = await page.screenshot({
    type: format,
    fullPage: true,
    omitBackground: format === "png" // Transparent background for PNG
  });
  
  return {
    buffer: screenshot,
    contentType: `image/${format}`,
    extension: format
  };
}
```

PNG captures full page height (scrollable content). JPG uses opaque background. Both maintain 1280px width.

### Context & Page Cleanup

```javascript
try {
  // ... rendering logic ...
} finally {
  await page.close();
  await context.close();
  releaseRenderSlot();
}
```

Critical: Always close page/context and release render slot, even on errors. Prevents memory leaks.

function escapeHtml(value: string) {

## Artifact Storage (exportArtifactStore.ts)

After rendering completes, the buffer (PDF/PNG/JPG) is stored for download:

**Storage Providers:**
1. **Local File System** - Dev/testing, stores in tmp/ folder
2. **AWS S3** - Production, via @aws-sdk/client-s3
3. **Hybrid** - Checks local first, falls back to S3

**Artifact Metadata:**
```javascript
{
  ownerType: "user" | "public-share",
  userId?: string,
  shareToken?: string,
  fileNameBase: "My Resume",
  contentType: "application/pdf",
  extension: "pdf",
  storageProvider: "local" | "s3",
  storageKey: "exports/uuid/resume.pdf",
  sizeBytes: 145234,
  expiresAt: "2026-04-29T10:00:00Z",
  bufferBase64?: string // For local storage
}
```

**Cache Duration:**
- Default 24 hours (configurable via EXPORT_ARTIFACT_TTL_MS)
- Stored in Redis with TTL
- User can re-download within window
- After expiry, job deleted and re-export required

**Idempotency:**
- Cache key includes SHA256 of snapshot + format + renderHtml
- Same snapshot = same output (bit-for-bit identical PDFs)
- Prevents redundant renders, improves performance

## Error Handling & Logging

**Error Types:**
- 400 Bad Request - Schema validation failed (Zod)
- 401 Unauthorized - Auth required
- 403 Forbidden - CORS or permissions denied
- 404 Not Found - Resource doesn't exist
- 409 Conflict - Export retry race condition
- 429 Too Many Requests - Rate limited
- 503 Service Unavailable - Export engine down (Playwright unavailable)
- 504 Gateway Timeout - Export took >30 seconds

**Structured Logging:**
All logs include:
- timestamp (ISO 8601)
- level (info, error, warn, debug)
- request_id (unique per request)
- userId (if authenticated)
- duration_ms (response time)
- status_code
- error details (stack trace on errors)

**Logging Middleware:**
Logs before and after request processing. Sensitive data (passwords, tokens) redacted.

## Performance Optimizations

1. **Browser Reuse** - Single Chromium instance vs launching new instance per render (orders of magnitude faster)
2. **Render Slot Semaphore** - Prevents memory exhaustion from too many concurrent renders
3. **Cache-Aside Pattern** - SHA256 deduplication prevents re-rendering identical snapshots
4. **Redis Session Cache** - Optional JWT strategy reduces database queries
5. **Connection Pooling** - Prisma handles via @prisma/adapter-pg
6. **Rate Limiting** - Prevents abuse, ensures fair resource distribution
7. **Lazy Font Loading** - Only loads Google Fonts stylesheet if needed (resolved from snapshot)
8. **Zero-Margin Rendering** - No whitespace around resume, maximizes ATS parser compatibility
