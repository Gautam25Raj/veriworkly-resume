# Building a Scalable Resume Platform

When VeriWorkly started, it was a single Next.js application holding everything: the resume builder, our blog, our documentation, and all the backend API routes. This monolithic approach was simple initially but quickly became unsustainable. Building the Docs meant recompiling the Resume Editor. Editing a Blog post slowed down Hot Module Replacement for critical API features.

Today, we're proud to run a massive, enterprise-grade **NPM Workspace Monorepo** encompassing **four entirely distinct applications**:

1. **Resume Builder** (pps/resume-builder)
2. **Docs Platform** (pps/docs-platform)
3. **Blog Platform** (pps/blog-platform)
4. **Backend Server** (pps/server)

Plus, everything is backed by our internal @veriworkly/ui design system.

Here's why we split the architecture and how it works.

## Welcome to the Monorepo

We manage the monorepo via **NPM Workspaces**. This setup provides significant advantages over the old monolith.

### 1. Isolated Build Pipelines

Now,
esume-builder is optimized specifically as a highly interactive Client-Side App, while log-platform and docs-platform are statically generated using **Fumadocs**. Next.js App Router caching policies apply strictly where they should. Modifying the documentation no longer restarts the Resume Editor.

### 2. Independent Deployments

Each application is containerized within a shared compose.yaml but runs as a separate service.

- **Resume Builder**: Port 3000
- **Docs Platform**: Port 3001
- **Blog Platform**: Port 3002
- **Backend API**: Port 8080

This isolation means that a heavy traffic spike hitting a viral log-platform article will never steal CPU cycles from someone trying to generate their PDF in the
esume-builder.

## Inside the Core Applications

### 🚀 Resume Builder: Local-First and Blazing Fast

Our editor (pps/resume-builder) is entirely local-first. We heavily lean into **Zustand** combined with the persist middleware.
When a user types their experience summary, the state immediately updates the DOM and seamlessly saves to localStorage. Only when the user hits "Sync" or "Download PDF" do we talk to our database or API.

### ⚙️ The Node.js Backend Server

The server (pps/server) handles two massive tasks that Next.js wasn't well suited for:

1. **Prisma & PostgreSQL Connection Pooling**: Direct connection to Neon Database with advanced cache strategies.
2. **Headless Chromium PDF Generation**: Generating pixel-perfect PDFs is mathematically intense. We chose not to use client-side libraries like jsPDF or html2canvas because they only create inaccessible images, which instantly fail ATS scans. Instead, the backend spawns **Playwright**, mounts a hidden React template over the API payload, waits for Tailwind 4 and fonts to render, and exports a raw, true-text PDF.

This heavy-lifting is moved entirely off the user's browser and isolated inside a robust Node 20+ Express instance running its own thread pool.

### 📚 The Docs Platform

Our documentation (pps/docs-platform) is an entirely independent app running Fumadocs on top of Next.js 15. The Docs Platform has a distinct openapi.yaml configuration to auto-generate our API reference, which would have absolutely bloated the Resume Editor if combined.

### ✍️ The Blog Platform

Finally, the Blog Platform (pps/blog-platform) is an SEO-optimized Markdown powerhouse. It shares the @veriworkly/ui package explicitly so buttons and cards match our main brand identity perfectly, but the Next.js configurations are heavily biased towards SSG (Static Site Generation), providing sub-100ms load times for global SEO dominance.

## Security & Authentication Deep Dive

To tie all these applications together under one security model, we use **Better-Auth**.
By storing session tokens centrally in the Neon database, our API endpoints at pps/server instantly know if a request originating from the Resume Builder is valid.

All passwords are cryptographically hashed using standard best-practices, but what's better is that **Better-Auth** handles magic links, Google, and Github SSO seamlessly without us maintaining standard OAuth boilerplate.

## Moving Forward and Tailwind CSS 4

By standardizing our @veriworkly/ui across all 3 frontends using **Tailwind CSS 4**, our design system is perfectly cohesive.

In closing, removing the monolithic architecture took significant time, but it resulted in a project that is infinitely easier to scale, debug, and maintain. Every piece of VeriWorkly now has one job, and it does that job perfectly. Welcome to the future of resume building.
