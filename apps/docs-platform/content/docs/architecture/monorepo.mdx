# Monorepo Architecture

VeriWorkly is organized as a single repository with multiple applications and one shared UI package.

The goal is straightforward: keep product areas independent (builder, docs, blog, backend) while still sharing core styles and components where it makes sense.

## The 4 Core Applications

### 1. Resume Builder (`apps/resume-builder`)

- **Port:** 3000
- **Framework:** Next.js (App Router)
- **Role:** Main product experience for creating and editing resumes
- **Tech highlights:** local-first state with Zustand + persist, template system, export/share flows

### 2. Node.js API Server (`apps/server`)

- **Port:** 8080
- **Framework:** Express + Node.js 20+
- **Role:** Auth, sync, sharing, export orchestration, and background processing
- **Tech highlights:** Prisma + PostgreSQL, queue-based export jobs, Playwright-powered PDF rendering

### 3. Docs Platform (`apps/docs-platform`)

- **Port:** 3001
- **Framework:** Next.js + Fumadocs
- **Role:** Product docs, architecture notes, and API reference
- **Tech highlights:** MDX content and generated OpenAPI artifacts

### 4. Blog Platform (`apps/blog-platform`)

- **Port:** 3002
- **Framework:** Next.js + Fumadocs
- **Role:** Product updates, engineering writeups, and career-focused content

## Shared Package (`packages/ui`)

`@veriworkly/ui` is an internal workspace package used across frontends for:

- Reusable primitives (buttons, cards, modals, menus, inputs)
- Shared container/layout utilities
- Font and theme variables

This keeps the look and behavior consistent without turning every app into one giant deploy unit.

## Why This Layout Works

- **Clear ownership:** each app has a clear responsibility and code boundary.
- **Faster iteration:** docs/blog updates do not require shipping resume-builder changes.
- **Operational flexibility:** services can be deployed and scaled independently.
- **Shared consistency:** common UI stays aligned through one internal package.
