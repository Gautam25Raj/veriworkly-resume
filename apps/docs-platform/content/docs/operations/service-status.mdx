# Checking Service Status

In a production environment, simply checking if the homepage loads is not enough. VeriWorkly relies on background workers (Playwright/BullMQ) and external databases.

This guide provides the exact workflows to monitor uptime, inspect resource constraints, and debug Docker container failures.

---

## 1. The Automated Health Endpoint

The most reliable way to monitor VeriWorkly is via the backend's automated health endpoint. This route actively probes the database and the Redis queue rather than just returning a static 200 response.

**Endpoint:** `GET /api/v1/health`

### Healthy Response (`200 OK`)

If all systems are nominal, the API will return:

```json
{
  "success": true,
  "message": "Server is healthy",
  "data": {
    "status": "ok",
    "database": "connected",
    "redis": "connected",
    "timestamp": "2026-04-26T10:00:00.000Z"
  }
}
```

### Unhealthy Response (`503 Service Unavailable`)

If there are issues with the database or Redis connection, the API will return:

```json
{
  "success": false,
  "message": "Service Unavailable",
  "data": {
    "status": "degraded",
    "database": "disconnected",
    "redis": "connected",
    "timestamp": "2026-04-26T10:05:00.000Z"
  }
}
```

<Callout type="info" title="Monitoring Integration">
  We highly recommend plugging `https://api.yourdomain.com/api/v1/health` into a monitoring tool
  like Datadog, UptimeRobot, or Better Stack. Configure the monitor to alert your team immediately
  if the status code deviates from 200.
</Callout>

## 2. Docker Container Diagnostics

If the health check returns a 503 or a container is stuck in a restart loop, you need to drop into the CLI to diagnose the issue.

### Check Container States

Ensure all defined services in your `compose.yaml` are running:

```bash
docker compose ps
```

If the `api` or `web` containers say `Exited (1)`, the Node process crashed.

### Monitor Resource Usage (OOM Kills)

Playwright PDF rendering can be memory-intensive. If background jobs are failing silently, your container might be running out of RAM and getting killed by the Docker daemon.

```bash
docker stats
```

Look at the `MEM USAGE / LIMIT` column. If the `api` container is hitting 100%, you need to provision more RAM to your host machine or limit concurrency in BullMQ.

## 3. Advanced Log Analysis

When a service crashes, the logs hold the answer. Run these commands from the directory containing your `compose.yaml`.

<Cards>
  <Card title="Tail Backend API" icon={<Terminal />}>

    ```bash
    docker compose logs --tail=100 -f api
    ```

    **What to look for:** Uncaught exceptions, Prisma connection errors, or missing environment variables.

  </Card>

  <Card title="Tail Redis Logs" icon={<Server />}>

    ```bash
    docker compose logs --tail=100 -f redis
    ```

    **What to look for:** `OOM command not allowed` (Redis ran out of memory) or background save failures.

  </Card>

  <Card title="Tail Next.js Frontend" icon={<Cpu />}>

    ```bash
    docker compose logs --tail=100 -f web
    ```

    **What to look for:** Fetch errors complaining about `ECONNREFUSED` (Frontend cannot reach the Backend internally).

  </Card>
</Cards>

## 4. Common Failure Modes

If you are experiencing downtime, check these common production pitfalls:

<Accordions>
  <Accordion id="redis-oom" title="Redis Memory Issues">
    If the Redis container is crashing or you see OOM errors in the logs, it means your queue is getting backed up and consuming too much memory.

    **Solution:** Consider increasing the RAM allocated to Docker or optimizing your BullMQ job processing limits.

  </Accordion>

  <Accordion id="db-conn" title="Database Connection Failures">
    If the API logs show repeated connection timeouts to PostgreSQL, ensure your `DATABASE_URL` is correct and that your database instance is healthy.

    **Solution:** Check for network issues or firewall rules blocking access to your external database (e.g., Neon).

  </Accordion>

  <Accordion id="startup-crash" title="API Crashes on Startup">
    If the API container fails immediately on startup, check for syntax errors in your `server/.env` file or missing environment variables.

    **Solution:** The logs will indicate which variable failed Zod validation.

  </Accordion>

  <Accordion id="cors-blocks" title="CORS Errors on the Frontend">
    If the UI loads but saving resumes fails silently or shows CORS errors in the browser console.

    **Solution:** Ensure the `FRONTEND_URL` in your `server/.env` exactly matches the URL in the browser (including `https://` and no trailing slashes).

  </Accordion>

  <Accordion id="playwright-zombies" title="PDF Generation Hanging">
    If BullMQ jobs are piling up but PDFs are not rendering, Playwright zombie processes might be hanging the worker thread.

    **Solution:** Restart the API container specifically by running `docker compose restart api`.

  </Accordion>
</Accordions>
