# Environment Variables

Running VeriWorkly in production requires strict configuration of environment variables. Because the architecture splits the frontend UI from the API and background workers, secrets must be routed to their respective contexts securely.

---

## 1. Frontend Configuration (`/.env`)

This file sits at the root of the project and configures the Next.js application.

| Variable                  | Requirement  | Description                                                                                                                                |
| :------------------------ | :----------- | :----------------------------------------------------------------------------------------------------------------------------------------- |
| `NEXT_PUBLIC_BACKEND_URL` | **Required** | The public-facing URL of the Express API. The browser uses this to make client-side requests. Example: `https://api.yourdomain.com/api/v1` |
| `NEXT_PUBLIC_APP_URL`     | Optional     | The canonical URL of your frontend, used for SEO metadata and absolute routing.                                                            |

---

## 2. Backend Configuration (`/server/.env`)

This file sits inside the `server/` directory. It contains highly sensitive infrastructure credentials. **Never commit this file to version control.**

### Core Infrastructure

<Cards>
  <Card title="Database (PostgreSQL)" icon={<Database />}>
    - `DATABASE_URL`: The full connection string to your PostgreSQL instance (e.g., Neon).
    - Ensure your string includes `?sslmode=require` if connecting to a managed cloud provider.
  </Card>

  <Card title="Cache & Queue (Redis)" icon={<Server />}>
    - `REDIS_URL`: The full connection string (e.g., `redis://localhost:6379`).
    - *Alternative:* If your host splits variables, use `REDIS_HOST`, `REDIS_PORT`, and `REDIS_PASSWORD`.
  </Card>
</Cards>

### Security & Authentication

<Cards>
  <Card title="CORS & Origin Control" icon={<Globe />}>
    - `FRONTEND_URL`: The exact URL of your Next.js application (e.g., `https://veriworkly.com`). The Express API uses this to enforce strict CORS policies and allow `HttpOnly` cookies.
  </Card>

  <Card title="Encryption Secrets" icon={<Shield />}>
    - `AUTH_SECRET`: A 32+ character string used by Better Auth to sign session cookies securely.
    - `JWT_SECRET`: A 32+ character string used for internal token signing and stateless verification.
  </Card>
</Cards>

### External Services (S3 & SMTP)

Because VeriWorkly generates PDFs via Playwright and handles user authentication, it requires storage and email routing.

<Cards>
  <Card title="AWS S3 (PDF Storage)" icon={<Cloud />}>
    - `AWS_ACCESS_KEY_ID`: Your IAM user access key.
    - `AWS_SECRET_ACCESS_KEY`: Your IAM secret key.
    - `AWS_REGION`: The bucket region (e.g., `us-east-1`).
    - `S3_BUCKET_NAME`: The destination bucket for rendered PDFs.
  </Card>

  <Card title="SMTP (Nodemailer)" icon={<Mail />}>
    - `SMTP_HOST`: Your mail provider (e.g., Amazon SES, Resend, SendGrid).
    - `SMTP_PORT`: Usually `587` or `465`.
    - `SMTP_USER` & `SMTP_PASS`: SMTP credentials.
    - `EMAIL_FROM`: The verified sender address (e.g., `noreply@veriworkly.com`).
  </Card>
</Cards>

## 3. Docker Overrides (`/.env.docker`)

When deploying via `docker compose`, the network topology changes. Containers cannot use `localhost` to talk to each other.

If you are using Docker, create `.env.docker` at the root. It should inherit your standard variables but **override the internal routing**:

```bash title=".env.docker"
# Tells Next.js SSR how to reach the Express API inside the Docker bridge network
BACKEND_INTERNAL_URL=http://api:8080/api/v1

# Tells the Express API how to reach Redis inside the Docker bridge network
REDIS_URL=redis://redis:6379
```

<Callout type="warn" title="Generating Cryptographic Secrets">
  Never use easily guessable strings for `AUTH_SECRET` or `JWT_SECRET`. To generate
  cryptographically secure, 32-byte hex strings suitable for production, run the following command
  in your terminal:
</Callout>
