# Docker Production Deployment

This guide explains how to self-host VeriWorkly in a production-like environment using Docker.

VeriWorkly utilizes a single root `compose.yaml` file to orchestrate the entire ecosystem, ensuring the Next.js frontend, the Express API, and the Redis cache all communicate securely over an internal Docker network.

<Callout type="warn" title="Important Database Note">
  **PostgreSQL is intentionally NOT containerized in this setup.** Because
  databases require complex volume management and backup strategies in
  production, our `compose.yaml` expects you to provide an external database URL
  (such as [Neon](https://neon.tech), AWS RDS, or a managed VPS Postgres
  instance).
</Callout>

---

## Prerequisites

- **Docker Engine** (version 24+)
- **Docker Compose** (version 2+)
- A valid external **PostgreSQL connection string**.

---

## Configuration

Before spinning up the containers, you must properly configure the environment variables so the Docker network can route traffic correctly.

<Steps>
  <Step>
    ### Copy the Environment Files
    There are three specific environment files you must create:

    ```bash
    # 1. Docker orchestration config
    cp .env.docker.example .env.docker

    # 2. Frontend & shared config
    cp .env.example .env

    # 3. Backend API config
    cp server/.env.example server/.env
    ```

  </Step>

  <Step>
    ### Configure Network Routing
    When running in Docker, internal containers talk to each other differently than the user's browser does. Pay strict attention to these variables in your `.env` files:

    * **`NEXT_PUBLIC_BACKEND_URL`**: This is what the *Browser* uses. It must be a publicly accessible URL (e.g., `https://api.yourdomain.com/api/v1` or `http://localhost:8080/api/v1`).
    * **`BACKEND_INTERNAL_URL`**: This is what Next.js uses during Server-Side Rendering (SSR) to talk to the API inside the Docker network. It should be set to `http://api:8080/api/v1` (where `api` is the service name in your compose file).

  </Step>
</Steps>

## Deployment

Once your `.env` files are populated with your database credentials and secrets, you can build and launch the application.

<Steps>
  <Step>
    ### Build and Start Services
    Run the following command from the root of the project. This will pull the Redis image, build the Next.js standalone output, build the Express backend, and start them in detached mode.

    ```bash
    docker compose --env-file .env.docker up -d --build
    ```

  </Step>

  <Step>
    ### Verify the Deployment
    Check the status of your running containers:

    ```bash
    docker compose ps
    ```

    You can watch the logs to ensure the backend successfully connected to your external database and the internal Redis container:

    ```bash
    # Watch all logs
    docker compose logs -f

    # Watch specific service logs
    docker compose logs -f api
    ```

  </Step>
</Steps>

## Production Recommendations

While `docker compose up` gets the application running, exposing port `3000` and `8080` directly to the internet is not recommended for a true production environment.

For a secure deployment, we highly recommend placing VeriWorkly behind a **Reverse Proxy / TLS Terminator**:

- **Nginx**
- **Traefik**
- **Caddy**
- **Cloudflare Tunnels**

The reverse proxy should handle your SSL certificates (HTTPS) and route traffic appropriately (e.g., routing `api.yourdomain.com` traffic to port `8080`, and `yourdomain.com` traffic to port `3000`).

---

## Managing the Deployment

If you need to update VeriWorkly, pull the latest code from GitHub and run the build command again:

```bash
# Pull latest code
git pull origin master

# Rebuild and restart containers without downtime
docker compose --env-file .env.docker up -d --build
```
